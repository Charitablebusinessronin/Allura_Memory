# ADR-001: Migrate from Custom Allura MCP to Team RAM + MCP Docker Skills Architecture

**Status:** Proposed | **Date:** 2026-04-21 | **Repo:** [Charitablebusinessronin/Allura_Memory](https://github.com/Charitablebusinessronin/Allura_Memory)

***

## Decision

Stop treating the custom Allura MCP as the execution, memory, routing, and session layer simultaneously. Promote **Team RAM** to the orchestration layer for **parallel agents and subagents** and demote MCP Docker servers to **focused skill containers**. **Allura Brain** remains the canonical, governed memory surface. Allura core retains policy, ADR logging, and Insight promotion — nothing else.

***

## Context & Problem

The current repo shows a `Dockerfile.mcp`, a `Dockerfile.mcp-server`, `mcp-entrypoint.sh`, and a `docker-compose.yml` that are all in the root — evidence that the custom MCP layer has been absorbing responsibilities incrementally.  The brittleness comes from **one surface doing five jobs**: transport, routing, memory recall, session management, and graph queries. Each additional capability added a new failure mode with no isolation boundary.

The `src/` directory and `.opencode/skills/` already contain much of the logic and scaffolding that should be split across focused skill containers instead of being centralized behind one custom MCP surface. The existing Docker infrastructure is an asset — it's already containerized and close to the shape of independent MCP skill servers. 

***

## New Architecture

```
┌─────────────────────────────────────────────┐
│              ALLURA CORE                    │
│  Policy engine · ADR log · Insight promotion│
│  (thinks, governs — does NOT execute)       │
└─────────────────────┬───────────────────────┘
                      │ task dispatch
┌─────────────────────▼───────────────────────┐
│            TEAM RAM ORCHESTRATOR            │
│  Parallel agents + subagents                │
│  Skill selection · context assembly         │
│  Retry logic · tool health tracking         │
└────┬──────────────┬──────────────┬──────────┘
     │              │              │
┌────▼───┐   ┌──────▼──────┐  ┌───▼────────┐
│Neo4j   │   │Cypher Query │  │Database    │
│Memory  │   │Skill        │  │Skill       │
│Skill   │   │(MCP Docker) │  │(MCP Docker)│
└────────┘   └─────────────┘  └────────────┘
     ↑ All reads/writes through retrieval API only
     ↑ No agent touches Neo4j directly
```

***

## Responsibility Boundaries

| Layer | Owns | Does NOT own |
|---|---|---|
| **Allura Core** | Policy, ADR log, Insight approval, Notion mirror | Routing, tool calls, session state |
| **Team RAM** | Skill selection, parallel dispatch, subagent coordination, retry, context assembly | Memory storage, graph writes, DB queries |
| **MCP Docker Skills** | Single-domain execution (Neo4j, Cypher, DB, search) | Cross-skill logic, orchestration, session, memory authority |
| **Raw Trace Store** | Append-only event log (Postgres/object store) | Knowledge promotion (curator owns that) |

### Canonical Surfaces

- **Allura Brain** is the canonical, governed memory surface for this repo.
- Use first-party `allura-brain_*` tools first for persistent memory operations.
- Use `MCP_DOCKER` only for approved discovery, configuration, activation, and lower-level governed access.
- Skill containers are focused execution surfaces, not agents.
- Team RAM orchestrates parallel agents and subagents; it does not become a second memory system.

## Migration Plan

### Phase 1 — Freeze the Custom MCP (This Week)
- Add a `// FROZEN: no new capabilities` comment block at top of the custom MCP entrypoint (`mcp-entrypoint.sh`, `Dockerfile.mcp`)
- File a GitHub issue: `[ARCH] Custom MCP feature freeze — migrating to skill model`
- Audit `docker-compose.yml` — identify which services map to Neo4j, Cypher, and DB skills already 

### Phase 2 — Extract Skill Containers (Week 2–3)
Create three standalone MCP Docker skill services from what already exists in `.opencode/skills/`:

```
.opencode/skills/
  skill-neo4j-memory/      # recall, store approved Insights
  skill-cypher-query/      # raw Cypher execution with result shaping
  skill-database/          # Postgres read/write for raw traces
```

Each skill exposes a minimal MCP tool surface (2–4 tools max). Each has its own `Dockerfile`, health check, and smoke test. Configure and activate them through `MCP_DOCKER_mcp-find`, `MCP_DOCKER_mcp-config-set`, and `MCP_DOCKER_mcp-add`.

### Phase 3 — Wire Team RAM Orchestration (Week 3–4)
Team RAM receives a task, selects skills, dispatches parallel agents and subagents, assembles context, and returns results. It never holds state — it reads approved Insights through **Allura Brain** and approved retrieval surfaces only.

```typescript
// team-ram/orchestrator.ts
const results = await Promise.allSettled([
  subagent.call('skill-neo4j-memory', { query: taskContext }),
  subagent.call('skill-cypher-query', { cypher: buildQuery(intent) }),
]);
const context = assembleContext(results);
```

### Phase 4 — Strip Allura MCP (Week 4–5)
Remove routing, session management, and direct DB calls from the custom MCP. What remains: policy validation, ADR writes, Insight promotion trigger. Move the rest to `archive/` (already exists in the repo).

## Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Skills share state via side channels | Medium | Enforce no shared volumes between skill containers |
| Team RAM retry storms if skill is down | Medium | Circuit breaker per skill, dead-letter queue |
| Insight approval queue grows stale | Low | Auto-escalate after 48h, ping in Notion |
| Cypher skill exposes raw graph write | High | Skill is read-only by policy; writes go through approval API only |

***

## What This Is NOT

- This is **not** a rewrite. The Docker infrastructure and `docker-compose.yml` are reused.
- This is **not** adding more complexity. It's moving existing complexity to the right layer.
- The custom MCP is not deleted — it's **stripped and narrowed**, then archived if unused.
